Component Scope — What Is Being Built

📍 How to Read This Section

Each component below lists every feature, module, API, and integration that must be built from scratch. This is not a screen count — it is an engineering scope. Use this section to evaluate whether the quoted price for each component is reasonable given what is actually being delivered. A component with 5 screens but 12 real-time systems, 3 payment flows, and live GPS is more expensive to build than a component with 30 simple static screens.

07A

Backend API Server — Full Engineering Scope

⚠️ Why the Backend Is the Most Expensive Component
The backend is not a simple REST API. It is three complete independent service platforms — Food Delivery (like Swiggy), Grocery Delivery (like Blinkit), and Home Services (like Urban Company) — all running on a single Node.js server with shared infrastructure but fully separate order engines, routing logic, financial systems, and real-time communication layers. Every feature visible in all 5 apps is powered entirely by this backend. There is no off-the-shelf solution for this combination. Every module listed below must be custom-engineered from scratch.
ModuleWhat It DoesWhy It Is Needed
🍽️ Food Order EngineFull order lifecycle: place → kitchen accept/reject → prep timer → rider assign → pickup → delivery → completion. Real-time status push via Socket.IO to customer, vendor, and rider simultaneously.Every food order placed on the Customer App flows through this engine. Without it, no food delivery works at all.
🛒 Grocery Order EngineSeparate engine with item-level picking checklist, out-of-stock substitution flow, pack-and-notify logic, grocery-specific delivery windows, and bulk item weight handling.Grocery orders are structurally different from food — items can be partially unavailable, weights vary, stores have minimum order rules. Cannot reuse the food engine.
🔧 Home Services Booking EngineTime-slot based booking with date/time selection, vendor capacity check, automatic or manual worker assignment, job status machine (booked → assigned → en route → arrived → in progress → extra charges → completed), rescheduling, and cancellation with policy enforcement.Home services use a completely different assignment model with no delivery rider. Jobs span hours and have multiple approval stages. Requires a separate engine from both delivery engines.
📍 GEO-Based Rider Auto-AssignmentWhen an order is ready for pickup, the system finds the nearest available online rider using Redis GEO commands, sends an assignment request with 25s countdown, handles rejection or timeout, cascades to next nearest rider, and logs full assignment history.Manual rider assignment is not scalable. Real-time GEO radius queries with cascading fallback are required for fast delivery operations. Uses Redis GEORADIUS + Socket.IO together.
🗺️ Real-Time Location TrackingBackground GPS location updates from Delivery Partner App and Worker App streamed to Redis (not database), then broadcast in real-time to the Customer App live tracking map via Socket.IO. Location data is ephemeral, refreshing every 5–10 seconds.Live map tracking in the Customer App requires a continuous location stream. Cannot be done with HTTP polling — requires Socket.IO + Redis pub/sub architecture.
💬 Socket.IO Real-Time LayerManages persistent WebSocket connections across all 5 apps simultaneously: order status updates, rider/worker location broadcasts, in-order chat messages, vendor incoming order alerts with sound, live admin dashboard data, and support ticket chat.Every real-time feature in every app routes through this layer. Must handle hundreds of simultaneous connections with room-based event routing (one room per active order).
💰 Wallet SystemCustomer wallet with Razorpay top-up flow, balance tracking, automatic deduction on order payment, refund credit, referral bonus credit, full ledger (every credit and debit recorded with source), and balance validation before checkout.The wallet is a financial system — every transaction must be atomic, auditable, and consistent. Requires database transactions and cannot lose data.
💳 Razorpay Payment IntegrationFull Razorpay integration: order creation, payment capture via SDK (UPI, card, net banking), webhook handling for success/failure/refund, refund initiation from admin, payout to vendor and rider bank accounts, payment status reconciliation.Every financial transaction — customer payments, vendor payouts, rider payouts, refunds — flows through Razorpay. Webhook handling must be secure (signature verification) and idempotent (no double-processing).
🏦 Commission & Payout EngineAutomatic commission calculation on every completed order (configurable % per service type, per-vendor override), net earnings calculation, payout scheduling, payout request handling, and full ledger of all commission deductions and payouts per entity.The platform earns revenue through commissions. Every order completion must trigger a calculation. Vendors and riders must see net earnings after commission. This is a financial sub-system.
📲 OTP Authentication (Fast2SMS)Phone OTP login for all 5 apps. OTP generation, Fast2SMS DLT-compliant delivery, 5-minute expiry, resend throttle (max 3 per hour), device session management, JWT token issuance and refresh.All 5 apps use phone OTP as the only login method. DLT registration is a legal requirement in India for commercial SMS. Session management must be secure across mobile and web.
🆔 KYC & Document VerificationDocument upload (FSSAI, GST, PAN, Aadhaar, Driving Licence, Vehicle RC) to AWS S3 via pre-signed URLs, admin review workflow (approve/reject with reason), re-upload after rejection, document expiry alerts, and a state machine (pending → under review → approved / rejected → re-submitted).Vendors and delivery partners cannot operate without verified documents. Regulatory compliance requires storing and verifying identity and business documents before onboarding.
🔐 RBAC — Role-Based Access Control5 distinct user roles (Customer, Vendor, Worker, Delivery Partner, Admin) each with completely separate API access. Admin sub-roles (Operations, Finance, Support) with granular per-module permissions. Every API endpoint validates both the auth token and the role permission before responding.A delivery partner must never access vendor earnings APIs. A support admin must not initiate payouts. RBAC enforced at the API layer is a security requirement, not optional.
🏷️ Coupons & Offers EngineCoupon creation and management (fixed or percentage discount, min order value, max discount cap, per-user usage limit, expiry date, service-specific scope), real-time validation at checkout, usage tracking, cashback credit to wallet, platform-wide offer banners.Promotions drive customer acquisition and retention. Coupon validation must happen server-side to prevent manipulation. Usage limits must be enforced atomically.
🎁 Referral SystemUnique referral code per customer, referral tracking (who referred whom), reward credit on first order by referred user, anti-abuse checks (same device or IP), referral history API.Referral programs are a primary growth channel for consumer apps. Requires server-side tracking to prevent gaming.
📊 Surge Pricing EngineConfigurable surge multiplier based on time-of-day or demand thresholds per zone. Applies automatically to delivery charges for food and grocery orders. Admin sets rules (e.g. 1.5× on weekends after 8pm in Zone A). Surge indicator shown to customer at checkout.Surge pricing is standard in delivery platforms. Requires a rule-evaluation engine that runs on every order creation to compute the correct delivery charge dynamically.
🗺️ Zone & Area ManagementAdmin-defined operational zones (polygon or pincode list), per-service zone config, customer location validation against active zones on every order, vendor delivery radius config, blackout zone support.Not all services are available everywhere. Zone validation on every order creation prevents orders from being placed outside serviceable areas.
🔔 Push Notifications (FCM)Firebase Cloud Messaging integration for all 5 apps. Triggered on every order status change, new order for vendor, new delivery for rider, new booking for worker, payout processed, support ticket reply. Batch notification API for admin broadcasts. Notification history stored per user.Without push notifications, users have no way to know their order status changed. FCM must be integrated on both the backend (trigger side) and all 4 mobile apps (receive side).
🕐 Cron Jobs & Scheduled TasksAuto-cancel orders not accepted within timeout, auto-complete orders after delivery timeout, payout batch processing, document expiry alerts, incentive campaign start and end, surge pricing schedule activation, daily revenue report generation.Many platform operations are time-triggered, not user-triggered. Without cron jobs, orders get stuck in intermediate states and payouts do not process automatically.
🎯 Incentive ProgramsAdmin-configurable delivery partner incentive campaigns (e.g. "Complete 10 deliveries today → ₹200 bonus"). Real-time progress tracking per rider, automatic bonus credit on completion, campaign validity enforcement, service-type specific campaigns.Incentives are how delivery platforms ensure rider supply during peak hours. Requires real-time order-count tracking per rider per active campaign.
🧾 Analytics & Reporting APIsServer-side aggregation for all Admin Dashboard analytics: daily/weekly/monthly revenue by service type, order volume trends, peak hour heatmaps, vendor performance metrics, rider utilisation, cancellation rates, repeat customer rates. CSV and Excel export endpoints.All Admin Dashboard analytics pages call these APIs. Complex aggregations on large datasets require optimised queries and potentially pre-computed summaries for performance.
🛡️ Security LayerJWT authentication with refresh token rotation, API rate limiting per user and IP, Razorpay webhook signature verification, S3 pre-signed URL expiry so documents are not publicly accessible, SQL injection prevention via Prisma ORM, CORS policy, security headers, input validation on all endpoints.A live financial platform with payments, personal data, and document storage is a high-value target. Each security measure listed prevents a specific class of attack.
📋 Audit LogEvery admin action (approve vendor, initiate refund, change commission, ban user, etc.) logged with timestamp, actor identity, action type, before-and-after values. Immutable — no delete allowed.Financial platforms require accountability. Disputes, fraud investigations, and compliance audits all depend on a reliable audit trail.
🗄️ Database Schema (PostgreSQL + Prisma)Full relational schema covering: users (5 types), orders (3 types), order items, vendors, menus and catalogues, services, bookings, workers, delivery partners, zones, wallets, transactions, payouts, commissions, coupons, referrals, notifications, support tickets, documents, audit logs, and more. Prisma migrations for version-controlled schema changes.The database schema is the foundation of the entire platform. An incorrectly designed schema causes data loss, performance problems, and expensive rewrites. This is significant design and implementation work.
⚡ Redis InfrastructureRedis used for: real-time rider and worker GPS location storage, Socket.IO adapter for multi-instance pub/sub, API response caching for high-traffic endpoints (kitchen listing, product catalogue), OTP storage with TTL, session data, and GEO commands for radius search.Without Redis, real-time location tracking is impossible at scale. GEO radius queries on PostgreSQL would be too slow for rider assignment under concurrent load.
07B

Admin Dashboard — Full Engineering Scope (40 Pages)

ℹ️ Why Admin Is Not Cheap Despite Being a Web Dashboard
The Admin Dashboard is a full Next.js web application — not a simple table viewer. It includes a live real-time map with rider and worker pins, 3-service analytics with charts and heatmaps, a complete financial management system (payouts, refunds, commissions), RBAC role management, configurable surge pricing and zone management, push notification broadcasting to all app users, and a full support ticket system with live chat. Many pages call complex aggregation APIs, require live Socket.IO data feeds, and handle financial operations with real consequences (initiating actual bank transfers, processing refunds via Razorpay). 40 pages of this nature is significantly more work than 40 marketing pages or a simple CRUD admin.
Module GroupPagesKey Technical Requirements
Live OperationsA-01 Overview Dashboard, A-02 Live Orders MapReal-time data via Socket.IO. Live order counts, GMV, active riders and workers auto-refresh. Full-screen Google Maps with moving rider/worker pins, order status overlays, filter controls. Requires persistent WebSocket connection and map marker clustering for performance at scale.

Infrastructure, DevOps & QA — Scope

Infrastructure & DevOps TaskWhat Is Set Up and Why It Matters
AWS EC2 + Nginx + PM2 clusterProduction server provisioning. Nginx reverse proxy routes /api to Node.js and / to Next.js admin panel. PM2 cluster mode uses all CPU cores for load distribution and auto-restarts crashed processes. Without this, the server uses only 1 CPU core and crashes are permanent until manually restarted.
GitHub Actions CI/CD pipelineAutomated test-and-deploy pipeline: every push to main branch triggers tests, build, and SSH deployment to EC2. Zero-downtime deploy strategy. Separate staging and production environments so bugs are caught before hitting real users.
PostgreSQL + Prisma migrationsProduction database setup, Prisma schema and full migration history, automated migration runs on each deploy, and database backup configuration via RDS snapshots or pg_dump cron. Without migrations, database changes require manual SQL runs that are error-prone and not version-controlled.
Redis setup (cache + pub/sub + GEO)Redis server configured for cache, Socket.IO multi-instance adapter, GEO radius commands, and OTP storage with TTL. Configured with AOF persistence so ephemeral location data survives restarts. Without Redis, real-time features and GEO assignment do not work.
SSL / HTTPS / Domain configurationLet's Encrypt SSL certificate, HTTPS enforcement (HTTP redirects to HTTPS), domain DNS configuration, www redirect. Required for Razorpay SDK to function and for App Store and Play Store approval.
